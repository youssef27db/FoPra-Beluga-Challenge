36class ExampleBelugaGymCompatibleDomain(BelugaGymCompatibleDomain):
37 """This is an example specialization of the BelugaGymCompatibleDomain class
38 which transforms PDDL-style states and actions from the original Beluga
39 scikit-decide domains to tensors to be used with deep reinforcement learning
40 algorithms. As shown at the bottom of this script, an instance of this class
41 must be passed to the BelugaGymEnv class, which will automatically populate
42 the gymnasium environment methods expected by deep reinforcement learning
43 solvers.
44
45 Please note that this class does not provide efficient tensor representations
46 for states and actions. It is only intended as exemplifying the transformation
47 from PDDL-style states and actions to tensors, and vice-versa.
48
49 State representation.
50 The PDDL-style state is composed of a set of predicates - and also fluents in the
51 case of the numeric PDDL encoding. Predicate and fluents take the form
52 (head obj1 ... objn), e.g. (at-side trailer1 fside). While a predicate can only be
53 true or false, a fluent can take a numeric value. For the tensor representation,
54 we treat predicates and fluents in the same way, assuming that a predicate is a fluent
55 taking Boolean values only. Our tensor representation is a fixed-size tensor,
56 but sufficiently large to handle problems of different sizes, using padding to disregard
57 the unused tensor entries. The observation space is a matrix NxM where each line
58 encodes a fluent and each column encodes the fluent's objects. Since fluents can
59 have a different number of arguments, we have as many columns as the the maximum
60 number of arguments across all the fluents, and we use padding to disregard entries
61 corresponding to unused arguments. It means that a fluent f, whose integer ID is f-id,
62 and whose arguments integer IDs are arg1-id, arg2-id, ..., and whose value if f-value,
63 is represented as a row vector in the form [f-id arg1-id arg2-id ... argn-id fvalue].
64 The arguments which are not used are equal to -1. For instance, if all the fluents
65 in the domain use at most 4 arguments, we would encode the Boolean predicate
66 (at-side trailer1 fside) taking a true value with a row vector in the form
67 [at-side-id trailer1-id f-side-id -1 -1 1].
68 IMPORTANT NOTE: in fact, PDDL-style states only enumerate Boolean predicates which
69 are true, for memory efficiency reasons. Our tensor representation does the same,
70 by listing only the rows corresponding to Boolean predicates that take a true value.
71 However, all the integer and float fluents are listed, as in PDDL.
72
73 Action representation.
74 The PDDL-style action takes the form of (head obj1 ... objn), e.g.
75 (get-from-hangar jig1 hangar1 trailer1). The tensor action is represented as a
76 1-dimensional tensor, i.e. a vector, in the form [action-id arg1-id ... argn-id]
77 where action-id, arg1-id, ..., argn-id are all integers. The number of argument
78 entries is equal to the maximum number of action arguments across all the actions
79 in the domain, meaning that unused argument entries for some actions are equal to -1.
80 For instance, if the maximum number of action arguments across all the actions in
81 the domain is equal to 5, the PDDL action (get-from-hangar jig1 hangar1 trailer1)
82 would be encoded as a vector [get-from-hangar-id jig1-ig hangar1-id trailer1-id -1 -1].
83
84 Applicable actions pitfall.
85 Whereas only a few actions are application in each possible given state, the number
86 of potential actions is huge (exponential in the number of action arguments). This
87 prevents from using standard masking techniques in deep reinforcement learning to
88 mask inapplicable actions in a given observation, because it would require first to
89 build a vector of size equal to the number of potential actions in the problem - which
90 is intractable for the Beluga problem. Finding a reasonable way to mask the inapplicable
91 actions in the Beluga environment is part of the challenge. In this simplistic
92 environment, we assume that all the actions are potentially applicable in each state,
93 but we penalize the inapplicable ones (see the reward signal description below). An
94 episode systematically ends when an action is applied in the current state where it is
95 not applicable.
96
97 Reward signal.
98 The objective of the Beluga environment is to find a policy which reaches a goal state
99 with a minimum number of steps. In this example environment, we propose to model the
100 reward signal as a function between 0 and 1, which is equal to 0 for all steps but the
101 terminal step which is equal to:
102 - 0 if the goal is not reached in the terminal step (meaning that this terminal step
103 corresponds to having applied an inapplicable action in the previous step or to having
104 exhausted the step budget)
105 - exp(-nb_of_steps) if the terminal step corresponds to a goal situation
106 That way, reaching the goal is always better than not reaching it, and it is better to
107 reach the goal with the fewest possible number of steps.
108 """
